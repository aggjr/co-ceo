CO-CEO · DER e Modelo de Operação de Dados

Documento técnico-operacional para consolidação do software de decisão CO-CEO. Mostra o DER das tabelas legadas e analíticas, além de como os dados são puxados do legado, transformados e publicados nas telas.

Escopo MySQL Legado + Schema ceo Artefatos JSON/JS das telas

Baseado nos scripts ativos do repositório: apollo_enterprise_miner.js, apollo_enterprise_engine.js, scripts/seed_ceo_ranked_skus.js, scripts/build_cd_purchase_plan.js, scripts/build_stock_health_histogram.js, scripts/legacy_live_bridge_server.js, scripts/build_legacy_ops_mapping.js.

1) Modelo Operacional (fim a fim)

1. Extração Legado

apollo_enterprise_miner.js

Puxa produto, movimentacao, tipomovimentacao, unidadenegocio, ativo e grava data/raw/sku_*.json.

2. Motor Apollo

apollo_enterprise_engine.js

Enriquece físico diário com ativoposicaoestoque (carry-forward) e gera bundles em data/js/sku_*.js.

3. Ingestão Analítica

seed_ceo_ranked_skus.js

Cria ranking e povoa schema ceo em daily_stock_snapshot + cd_daily_aggregate (com engine_run).

4. Produtos Analíticos

build_cd_purchase_plan.js, build_stock_health_histogram.js, build_top_important_skus.js

Combinam legado + ceo + bundles e publicam JSON/JS para telas.

5. Consumo em Tela

ceo_product_detail_layout.html, cd_purchase_plan.html, stock_health_histogram.html, apollo_grid.html.

Bridge live opcional: legacy_live_bridge_server.js para D-1 e histórico físico por SKU.

2) DER · Domínio Legado (origem)

Regra central: ativo é a ponte entre produto e unidadenegocio. Snapshots físicos: ativoposicaoestoque (histórico) e ativototalizador (estado atual).

3) DER · Camada Analítica CO-CEO (schema ceo)

Observação: no schema atual as relações com run_id são de integridade lógica (não há FK física definida). Recomendação para produto: ativar FKs e políticas de retenção por execução.

6) Artefatos não-SQL que compõem o produto

  • data/raw/sku_*.json — extração bruta por SKU (movimentação por unidade).
  • data/js/sku_*.js — bundle operacional por SKU (timeline + métricas por unidade).
  • data/client/network_matrix.json — matriz de sugestão SKU × loja para plano CD.
  • data/client/cd_purchase_plan.json/.js — priorização e tomada de decisão CD.
  • data/client/stock_health_histogram.json/.js — saúde de estoque (quantidade e capital).
  • data/client/sku_top_important.json/.js — ranking de SKUs prioritários.

7) Recomendações para virar software produto (próximo passo)

  1. Formalizar FKs físicas no schema ceo com política de retenção por engine_run.
  2. Criar dicionário de dados versionado (campo, tipo, regra de negócio, origem).
  3. Separar camadas em serviços: ingestão, motor, publicação API, front-end.
  4. Adicionar observabilidade: latência, volume por tabela, cobertura de SKU, alertas de desvio.
  5. Publicar contratos estáveis para telas (OpenAPI/JSON schema para datasets).
